# mr-review — Fetch MR threads and write a resolution plan

Fetch all open discussion threads from a GitLab MR, infer logical
changesets, and write `.claude/mr-plan.md` for the user to edit before
running `/mr-resolve`.

## Step 1 — Parse the URL

`$ARGUMENTS` is a full GitLab MR URL, e.g.:
`https://gitlab.com/group/project/-/merge_requests/42`

Parse it to extract:
- `$MR_URL` — the full URL as given (stored verbatim in the plan)
- `$HOST` — e.g. `gitlab.com`
- `$PROJECT_PATH` — everything between the host and `/-/merge_requests/`,
  e.g. `group/project` or `group/subgroup/project`
- `$IID` — the numeric segment after `/merge_requests/`, e.g. `42`
- `$ENCODED_PATH` — `$PROJECT_PATH` with every `/` replaced by `%2F`,
  e.g. `group%2Fproject`

If `$ARGUMENTS` is empty, attempt auto-detection:
```bash
glab mr view --output json 2>/dev/null
```
If that succeeds, extract `iid` and `web_url` from the JSON and derive
the fields above from `web_url`. If it fails, ask the user to provide
a URL and stop.

## Step 2 — Fetch MR metadata

```bash
glab mr view $IID --output json
```

Record from the response:
- `iid` (confirm it matches `$IID`)
- `title`
- `source_branch`
- `target_branch`
- `web_url`

## Step 3 — Fetch all discussion threads

```bash
glab api "projects/$ENCODED_PATH/merge_requests/$IID/discussions"
```

For each discussion object in the array:
- **Skip** if `individual_note` is `true`
- **Skip** if `notes[0].system` is `true`
- **Skip** if `notes[0].resolved` is `true`
- **Keep** all others (unresolved, human-authored threads)

For each kept thread record:
- `discussion_id` — the top-level `id` field (UUID, needed for API replies)
- `note_id` — `notes[0].id` (human-visible `#NNN` reference)
- `author` — `notes[0].author.username`
- `body` — `notes[0].body` (full text, do not truncate)
- `file_path` — `notes[0].position.new_path` if present, else `null`
- `line` — `notes[0].position.new_line` if present, else `null`
- `replies` — any entries in `notes[1..]` for context (author + body)

## Step 4 — Infer changesets

Group threads into named changesets using these heuristics in order:

1. **Same file, adjacent lines** — threads on the same file within ~20
   lines of each other address the same code. Group them.
2. **Thematic similarity** — threads whose bodies share a concern
   (error handling, naming conventions, test coverage, types, docs)
   should be grouped even across files.
3. **Explicit cross-references** — honour "same issue as above" or
   "also here" when a reviewer says so.
4. **No forced groupings** — if a thread has no clear relationship to
   others, give it its own changeset rather than grouping arbitrarily.

Name each changeset with a short verb-noun title.
Assign each a sequential letter: A, B, C, …

## Step 5 — Write the plan file

Write `.claude/mr-plan.md` using exactly the structure below.
Do not truncate thread bodies. The `Discussion ID` and `Note ID` lines
are both required — `/mr-resolve` uses the UUID for API calls and the
note ID for human-readable references and commit messages.

```markdown
# MR Plan: !$IID — "$TITLE"

<!-- Generated by /mr-review. Edit freely before running /mr-resolve.  -->
<!-- Delete a changeset block to skip it. Reorder blocks to change     -->
<!-- commit sequence. Do not edit the metadata block.                  -->

## Metadata
<!-- Do not edit this block — /mr-resolve reads it programmatically -->
- **MR URL:** $MR_URL
- **IID:** $IID
- **Project path:** $PROJECT_PATH
- **Encoded path:** $ENCODED_PATH
- **Source branch:** $SOURCE_BRANCH
- **Target branch:** $TARGET_BRANCH

---

## Changeset A: $TITLE
- **Status:** pending
- **Note IDs:** #$NOTE_ID, #$NOTE_ID
- **Discussion IDs:** $DISCUSSION_UUID, $DISCUSSION_UUID
- **Files:** $FILE_PATH
- **Approach:** $PARAGRAPH — describe what the reviewers are asking for,
  why, and the specific approach planned. Be concrete: mention function
  names, patterns, or utilities where relevant.

### Threads
- **#$NOTE_ID** (@$AUTHOR) `$FILE:$LINE`
  > $FULL_BODY

- **#$NOTE_ID** (@$AUTHOR) `$FILE:$LINE`
  > $FULL_BODY

---

## Changeset B: $TITLE
- **Status:** pending
...
```

### Status field

The `Status` field is machine-read by `/mr-resolve` to track progress
across interrupted or iterative runs. Valid values:

| Value | Meaning |
|---|---|
| `pending` | Not yet started |
| `committed ($SHA)` | Changes committed locally, reply not yet posted |
| `done ($SHA)` | Committed, pushed, and thread reply posted |

Write every changeset as `pending` when creating the plan. Do not
modify status values — `/mr-resolve` owns them.

## Step 6 — Report to the user

After writing the file, tell the user:
- How many open threads were found
- How many changesets were inferred, with their titles
- The path to the plan file
- That they should review the groupings, edit the Approach paragraphs,
  delete any changesets they want to skip, and then run `/mr-resolve`

Do not begin making any code changes.
